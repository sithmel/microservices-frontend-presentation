Frontend integration

We took the architecture decision of exposing the frontend of a series of microservices in the browser.

Frontend integration

We are able to do this thanks to compoxure.

Compoxure

Compoxure is an http proxy that provides the integration of many microservices merging all fragments coming from them in a single html page. It works this way: a request is routed to a microservice passing through compoxure

Compoxure

the microservice returns a page, where the secondary content is missing. And instead is marked by special tags.

Compoxure

These cx tags instruct compoxure to get fragments from different microservices The page is composed and delivered to the browser

Compoxure

performance and fallback

Compoxure manages also the caching of these fragments. It also tries to fallback to a stale cached version of a service rather than fail if the microservice is down.

So here it is, services are well isolated each other. And compoxure is taking care of both performance (with caching) and reliability (fallback). If only there was no client side javascript! Yes, because the issue as soon as the content reaches the browser is having more than one application coexisting and also cooperate with the others.

A good solution should:

  • be performant!
  • keep services independent
Before explaining all solutions I would like to explain some design principle I used to tell if a solution was good or not We can't propose a solution that harm the performance or the SEO. Mobile performance are vital for a global businness. We need to value our user's time! Indipendence is very important for microservices. It allows: * develop easily locally * deploy independently * reuse different services in different environments/pages

A different approach

async-define

async-define is a AMD runner, someone of you probably used requirejs. Requirejs used AMD as a system to define a module, If you have used requirejs you know that bundling many amd modules together, in the right order is necessary. Every time a new dependency is required requirejs triggers a download. This may lead to a waterfall of downloads. Async-define avoided completely this issue. async-define uses AMD to synchronise the execution of multiple scripts. You, as a web author have to put all required script tags in the page. Async-define manages only the execution order.

async-define: define your dependencies


            var React = require('react');
            var ReactDOM = require('react-dom');
            var asyncDefine = require('asyncDefine');

            asyncDefine('react-16', function (){
              return React;
            });

            asyncDefine('react-dom-16', function (){
              return ReactDOM;
            });

async-define: use your dependencies


            var asyncDefine = require('asyncDefine');

            asyncDefine(['react-16', 'react-dom-16'],
            function (react, reactDOM){
              ... your code runs here ...
            });

For increasing the independence I also advise to load the common bundle once for every service. It may look as a waste, but you should take into consideration that: * async-define prevents a dependency to be executed more than once * many browsers already load the same resource only once * compoxure prevents this when using the cx-bundles/cx-library

Issue 3

Microservices communication

Last but not least, how can different applications can communicate each others? The main point here is that every application should be able to run independently, and not be tightly coupled to others. The easiest way is to introduce a pub/sub library in global namespace of your page.

Solution: use an event bus

The easiest way is to introduce a pub/sub library in global namespace of your page.

occamsrazor

Expose the bus in the global namespace


						const or = require('occamsrazor');
						window.eventBus = or();

occamsrazor

Subscribe an event


						window.eventBus.on('login', (event, obj) => {
              ... do something when logged in ...
						});

occamsrazor

trigger an event


						window.eventBus.trigger('login', { username: 'Maurizio' });

occamsrazor

  • lightweight
  • ton of features
  • broadcast across windows/iframes/tabs
  • can bridge history API (pushstate, popstate)
